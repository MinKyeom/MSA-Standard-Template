# 로컬 실행 및 GitHub 자동 배포 가이드

이 문서만 따라 하면 **로컬에서 전체 MSA 스택을 띄워 확인**한 뒤, **`main`(또는 `master`) 브랜치에 push 하면 CI가 이미지를 빌드해 GHCR에 올리고**, 설정해 둔 **서버에서 자동으로 pull 후 재기동**하는 흐름을 재현할 수 있습니다.

**GitHub에 공개 저장소를 새로 만들고 CI/CD를 처음 연결하는 절차**는 [GitHub-공개저장소-CI-CD-가이드.md](./GitHub-공개저장소-CI-CD-가이드.md)를 참고하세요.

---

## 1. 전제 조건

| 항목 | 설명 |
|------|------|
| Docker | **Docker Desktop**(Mac/Win) 또는 Linux에서 **docker 엔진 + 데몬 기동** |
| Compose | **Compose V2** — 명령은 반드시 **`docker compose`**(공백, 하이픈 없음). 구버전 `docker-compose`(하이픈)는 이 저장소 스크립트에서 쓰지 않습니다. |
| Git | 저장소 클론 가능 |
| 메모리 | 전체 스택 기준 **약 4GB 이상** 여유 권장(기존 compose 주석과 동일) |
| 운영 서버 | Ubuntu 등 Linux, Docker / Git 설치, 프로젝트 디렉터리가 Git 원격과 연결됨 |

### 1.1 Mac에서 꼭 확인할 것

1. **Docker Desktop 앱을 실행**합니다(메뉴바에 고래 아이콘, “Docker Desktop is running”).
2. 터미널에서 다음이 **에러 없이** 나와야 합니다.

```bash
docker info
docker compose version
```

`failed to connect to the docker API` / `docker.sock: no such file` 가 나오면 **데몬이 꺼져 있는 것**입니다. Docker Desktop을 켠 뒤 다시 시도하세요.  
(Colima/Lima만 쓰는 경우 해당 도구로 엔진을 띄운 뒤 `docker context`가 그 소켓을 가리키는지 확인합니다.)

3. 컨테이너 올리기(수동 예시 — `-d` 대신 **`--detach`** 권장):

```bash
docker compose up --detach --build
```

Mac **Apple Silicon**에서 AWS Lightsail **amd64**용 이미지를 로컬에서 만들 때는 터미널에서 한 번 실행합니다.

```bash
export DOCKER_DEFAULT_PLATFORM=linux/amd64
```

일반적인 로컬 테스트(Mac에서 Mac용 컨테이너)에서는 생략해도 됩니다.

## 2. 로컬에서 한 번에 실행하기

### 2.1 최초 설정

1. 저장소 루트로 이동합니다.
2. 환경 파일을 만듭니다.

```bash
./scripts/init-env.sh
```

또는 수동으로:

```bash
cp .env.example .env
```

3. **`.env`를 열어** 최소한 아래를 맞춥니다.

- **`JWT_SECRET`**: 개발용이라도 긴 임의 문자열(운영과 동일 규칙).
- **`POSTGRES_PASSWORD`**: 로컬 전용이면 `.env.example`의 예시대로 두거나 변경.  
  **이미 데이터 디렉터리(`data/`)가 있는데 비밀번호만 바꾸면 DB 접속이 안 됩니다.** 기존 DB를 쓰는 경우 예전 비밀번호와 동일하게 유지하세요.
- **`GROQ_API_KEY`**: 챗봇을 쓰려면 실제 키. 없으면 해당 기능만 실패할 수 있습니다.

**로컬 전용 이미지 이름**은 스크립트가 기본으로 고정합니다.

- `scripts/local-up.sh` 실행 시 셸에서 `DOCKER_IMAGE_PREFIX=local/msa`, `IMAGE_TAG=latest`를 export 하므로, `.env`에 `ghcr.io/...`가 있어도 **로컬 빌드 이미지**가 우선됩니다(Compose 규칙: 셸 환경 변수가 `.env`보다 우선).

### 2.2 빌드 및 기동

```bash
./scripts/local-up.sh
```

내부 동작은 다음과 같습니다.

- `scripts/compose.sh` → 내부에서 **`docker compose`만** 호출 (V2 플러그인)
- `build --parallel` 후 `up --detach --build` (`-d` 대신 `--detach` — 일부 CLI에서 오류 방지)

### 2.3 접속 확인

| 용도 | URL |
|------|-----|
| **웹 UI** | http://localhost:3000 ← 브라우저는 여기로 접속 |
| API Gateway | http://localhost:8085 |
| 헬스(예) | http://localhost:8085/actuator/health |

**참고:** 게이트웨이 루트 `http://localhost:8085/` 는 라우트가 없어 **404가 정상**입니다. API는 `/api/posts`, `/auth/...` 처럼 경로를 붙이거나, 화면은 **3000번**을 사용하세요.

로그 확인:

```bash
bash scripts/compose.sh logs -f api-gateway
bash scripts/compose.sh ps
```

### 2.4 중지

```bash
./scripts/local-down.sh
```

## 3. 프론트만 호스트에서 개발하는 경우(선택)

Docker 대신 Next dev 서버를 쓰려면 `frontend/nextjs-app`에서:

```bash
cp ../../.env.example .env.local   # 또는 수동으로 NEXT_PUBLIC_* 만 작성
npm install
npm run dev
```

`.env.local`의 `NEXT_PUBLIC_API_URL` 등은 **브라우저가 접근 가능한 주소**(보통 `http://localhost:8085`)로 맞춥니다. 백엔드는 위 Docker 스택으로 띄운 상태를 권장합니다.

## 4. Git push → 서버 자동 배포(CI/CD)

### 4.1 동작 요약

1. **`main` 또는 `master`**에 push(또는 Actions에서 **workflow_dispatch** 수동 실행).
2. **GitHub Actions**가 `docker compose build` 후 이미지를 **GHCR**(`ghcr.io/<소문자 owner>/<소문자 repo>/...`)에 푸시합니다. 태그는 **커밋 SHA**입니다.
3. **Secrets가 모두 채워져 있으면** SSH로 서버에 접속해 `git pull` 후 `IMAGE_TAG=<해당 SHA>`로 `docker compose pull` · `up -d` 합니다.

Secrets가 하나라도 비어 있으면 **배포 job은 건너뛰고**, 빌드·푸시만 수행됩니다.

### 4.2 GitHub 저장소 Secrets

| Secret 이름 | 설명 |
|-------------|------|
| `ENV_VARS` | 로컬 `.env` 와 동일한 **다줄** 내용 |
| `HOST` | 서버 IP 또는 도메인 |
| `USERNAME` | SSH 로그인 사용자 |
| `KEY` | SSH 개인 키 전체(PEM) |
| `DEPLOY_PATH` | (선택) git 루트 절대 경로. 비우면 `~/저장소이름` |

`HOST`·`USERNAME`·`KEY`·`ENV_VARS` 가 비어 있으면 배포 job은 실행되지 않습니다. 빌드도 `ENV_VARS` 가 필요합니다.

### 4.3 서버 쪽 필수 준비

1. **같은 저장소를 클론**하고, `DEPLOY_PATH`와 동일한 경로를 씁니다.
2. 서버 루트에 **`.env`**를 두고, 아래를 **반드시** CI와 맞춥니다.

```env
# GHCR 경로는 GitHub 사용자명/저장소명과 동일하고 소문자
DOCKER_IMAGE_PREFIX=ghcr.io/깃허브사용자명/저장소명
```

`IMAGE_TAG`는 워크플로에서 매 배포마다 `github.sha`로 export 하므로 **서버 `.env`의 IMAGE_TAG는 덮어써집니다.**

3. 서버에 **Docker**와 **Git**이 있어야 합니다.
4. **GHCR 이미지가 private**인 경우, 서버에서 한 번 로그인합니다.

```bash
echo "YOUR_GITHUB_PAT" | docker login ghcr.io -u GITHUB_USERNAME --password-stdin
```

Public 패키지면 로그인 없이 pull 됩니다.

5. 배포 사용자가 해당 디렉터리에서 `git pull`, `docker compose`를 실행할 수 있어야 합니다(권한·sudo 정책 확인).

### 4.4 운영용 프론트 빌드 URL

워크플로는 **`ENV_VARS` Secret** 안의 `NEXT_PUBLIC_*` 로 Docker 빌드 인자를 채웁니다. 별도 Repository Variables 는 필수 아님.

## 5. Nginx(호스트)와 Fluent Bit

- **Nginx**: `/etc/nginx` 에 올리는 절차·CI와의 관계는 [Nginx-운영-가이드.md](./Nginx-운영-가이드.md) 참고. 이식용은 `nginx/msa-project.conf.template` + `scripts/render-nginx.sh` + `nginx/.env.nginx.example` 입니다(렌더 결과 `nginx/msa-project.rendered.conf` 는 `.gitignore`).
- **Fluent Bit**: Compose 서비스 `fluent-bit`는 **`monitoring` 프로필**에만 포함됩니다. GitHub Actions 배포는 `COMPOSE_PROFILES=edge`만 쓰므로 **배포 시 자동으로 뜨지 않습니다.** 서버에서 필요할 때 `bash scripts/start-fluent-bit.sh`(또는 `docker compose --profile monitoring up -d fluent-bit`)로 기동하고, 중지는 `bash scripts/stop-fluent-bit.sh` 또는 `docker compose stop fluent-bit`입니다. 항상 앱과 함께 켜려면 `.env`에 `COMPOSE_PROFILES=edge,monitoring`을 넣을 수 있습니다. `LOKI_HOST` / `LOKI_PORT`로 원격 Loki를 가리킵니다.

## 6. 자주 묻는 문제

| 증상 | 확인 |
|------|------|
| `unknown shorthand flag: 'd' in -d` (`docker compose up -d` 시) | `docker compose`가 플러그인으로 안 붙은 상태에서 인자가 꼬일 때가 있습니다. **`docker compose up --detach --build`** 로 실행하세요. |
| `docker.sock: no such file` / `failed to connect to the docker API` | **Docker 데몬이 안 떠 있음**입니다. Docker Desktop 실행 후 `docker info` 로 확인. `docker-compose`(하이픈)만 깔려 있어도 데몬이 없으면 동일 오류가 납니다. |
| `env_file .env` 오류 | `./scripts/init-env.sh`로 `.env` 생성 |
| DB 연결 실패 | `POSTGRES_PASSWORD`가 기존 `data/*_db` 생성 시와 같은지 |
| CORS 오류 | `.env`의 `CORS_ALLOWED_ORIGINS`에 프론트 Origin(예: `http://localhost:3000`) 포함 |
| 서버 `pull`은 되는데 옛 이미지 | `DOCKER_IMAGE_PREFIX`가 GHCR 경로와 **소문자로** 일치하는지 |
| 배포 job이 안 돌음 | `HOST`·`USERNAME`·`KEY`·`ENV_VARS` 설정 여부, 브랜치가 `main`/`master`인지 |
| `eclipse-temurin:17-jre-alpine` / `no match for platform in manifest` | Apple Silicon 등 **arm64**에서 Alpine JRE 태그가 없을 때 발생. 저장소 Dockerfile은 **`eclipse-temurin:17-jre-jammy`** 로 맞춰 두었습니다. 다시 `docker compose build --no-cache` 후 `up` 하세요. |
| `Failed to load SWC binary for linux/arm64` / `__res_init: symbol not found` | **node:alpine**(musl) 컨테이너 안에서 Next SWC가 깨질 때가 많습니다. 프론트 Dockerfile은 **`node:18-bookworm-slim`**(glibc)을 씁니다. 프론트만 재빌드: `docker compose build --no-cache frontend` |
| 로컬에서 8085·API만 404 / 화면이 비정상 | **①** UI는 **http://localhost:3000** 인지 확인. **②** `http://localhost:8085/` 단독은 404가 맞음. **③** Docker에서 프론트 SSR이 `localhost:8085`로 붙으면 실패하므로 compose에 **`GATEWAY_INTERNAL_URL=http://api-gateway:8085`** 가 들어가 있어야 함(저장소 기본값). 변경 후 `docker compose up -d --build frontend` |

## 7. 관련 파일

| 경로 | 역할 |
|------|------|
| `docker-compose.yml` | 전체 서비스 정의 |
| `.env.example` | 변수 목록(복사 후 `.env`) |
| `scripts/init-env.sh` | `.env` 자동 생성 |
| `scripts/compose.sh` | `docker compose`만 호출(데몬·플러그인 검사) |
| `scripts/local-up.sh` / `local-down.sh` | 로컬 기동·중지 |
| `.github/workflows/deploy.yml` | 빌드·푸시·SSH 배포 |
| `docs/GitHub-공개저장소-CI-CD-가이드.md` | 공개 저장소 생성·Secrets·Variables·서버 연결 |
| `docs/배포-등록-샘플.md` · `docs/templates/*` | 배포용 빈 양식 |
| `nginx/msa-project.conf.template` | Nginx 치환용 템플릿 |
| `docs/Nginx-운영-가이드.md` | `/etc/nginx` · render-nginx · CI 관계 |

이 가이드와 저장소 설정이 **로컬 검증 → Git push → 서버 반영** 구조를 만족하는지 주기적으로 확인하세요.
